# Antigravity 스킬 & Bkit 방법론 사용 가이드

이 가이드는 고품질 소프트웨어 개발을 보장하기 위해 Antigravity(Bkit Vibecoding Kit)에 통합된 스킬과 패턴을 효과적으로 사용하는 방법을 설명합니다.

## 1. 핵심 철학: PDCA (Plan-Do-Check-Act)

가장 중요한 규칙은 **PDCA 사이클**을 따르는 것입니다. Antigravity는 작업 규모에 따라 이 단계를 자동으로 안내하도록 설계되었습니다.

| 단계 | 명령어 | 목표 |
| :--- | :--- | :--- |
| **P**lan (계획) | `/pdca plan {기능명}` | *무엇을*, *왜* 만들지 정의합니다. |
| **D**esign (설계) | `/pdca design {기능명}` | *어떻게* 만들지 정의합니다 (아키텍처, 스키마, API). |
| **D**o (실행) | `/pdca do {기능명}` | TDD를 따라 코드를 구현합니다. |
| **C**heck (검토) | `/pdca analyze {기능명}` | 갭 분석: 코드가 설계와 일치하는지 확인합니다. |
| **A**ct (개선) | `/pdca iterate {기능명}` | 분석 중 발견된 문제를 수정합니다. |
| **Report** (보고) | `/pdca report {기능명}` | 수행한 작업의 최종 요약을 생성합니다. |

> [!IMPORTANT]
> **추측 금지**: 확실하지 않으면 문서를 확인합니다. 문서에 없으면 사용자에게 질문합니다.
> **SoR (단일 진실 공급원)**: 코드베이스 > 문서 > 설계 문서 순서로 우선합니다.

---

## 2. 기술 수준 및 행동 방식

Antigravity는 프로젝트 수준을 감지하여 응답 스타일을 조정합니다:

- **Starter (초급)**: 간단한 정적 사이트. 상세하고 친절한 설명을 제공합니다.
- **Dynamic (중급)**: DB/Auth(BaaS)를 사용하는 풀스택. 기술적이지만 명확한 로직으로 설명합니다.
- **Enterprise (고급)**: 마이크로서비스, K8s, Terraform. 간결하고 아키텍처 중심으로 설명합니다.

---

## 3. 전문 유틸리티 스킬

특정 문제에 직면했을 때 다음 전문 스킬을 사용합니다:

### 🔬 시스템적 디버깅 (Systematic Debugging)
**대상**: 버그, 테스트 실패, 예상치 못한 동작.
- **철칙**: 근본 원인 조사 없이는 수정하지 않습니다.
- **프로세스**: 로그 읽기 → 재현 → 증거 수집 → 가설 수립 → 최소 수정.
- **경고 신호**: 3회 이상 수정이 실패하면 반드시 멈추고 아키텍처를 의심해야 합니다.

### 🧪 테스트 주도 개발 (TDD)
**대상**: 모든 기능 개발 및 버그 수정.
- **철칙**: 실패하는 테스트 없이는 프로덕션 코드를 작성하지 않습니다.
- **사이클**: RED (실패 테스트 작성) → GREEN (통과할 최소 코드) → REFACTOR (정리).

### 📝 계획 작성 (Writing Plans)
**대상**: 복잡하고 여러 단계로 구성된 작업.
- **접근 방식**: 2-5분짜리 "한입 크기" 작업으로 분해합니다.
- **실행**: 빠른 반복을 위해 서브 에이전트가 처리할 수 있습니다.

## 4. 사용자(당신)를 위한 모범 사례

저(Antigravity)를 최대한 활용하려면:

1.  **용어를 먼저 정의하세요**: **phase-1-schema**를 사용하여 함께 사용할 용어를 정의합니다.
2.  **컨벤션을 설정하세요**: **phase-2-convention**을 사용하여 코딩 스타일을 정의합니다 (예: "절대 경로 사용", "특정 패턴으로 에러 처리").
3.  **설계를 검증하세요**: 코드를 한 줄이라도 작성하기 전에 `/pdca design` 결과물을 검토합니다.
4.  **TDD를 강제하세요**: 제가 코딩에 바로 뛰어드는 것 같으면 "테스트를 먼저 작성해"라고 상기시켜 주세요.
5.  **갭 분석을 활용하세요**: 기능을 완료한 후 "갭 분석"을 요청하여 누락된 것이 없는지 확인합니다.

---

## 5. 작업 경계 및 이력 기록

전역 규칙에 따라 저는 항상:
1.  **이력 기록**: `YYYYMMDD HH:mm:SS: [작업 요약]` 형식으로 `GEMINI.md` 또는 `task.md`에 기록합니다.
2.  **작업 경계 사용**: 현재 진행 상황(계획 → 실행 → 검증)을 명확하게 유지합니다.
